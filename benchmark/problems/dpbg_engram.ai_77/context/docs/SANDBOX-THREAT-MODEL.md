# Sandbox Threat Model — `activelearning-sandbox`

> Scope: the container the **Meta-Programmer** uses to execute and test
> LLM-generated code. Image: `activelearning-sandbox:latest`
> (`sandbox/Dockerfile`). Runtime spawner:
> `meta-programmer/src/meta_programmer/sandbox_manager.py`.
> Tracking issue: [#65](https://github.com/DPBG/Engram.AI/issues/65) (E1.3.1).

## 1. What we are defending against

Engram can write its own code (`meta-programmer/`). That code is **untrusted by
construction**: it is generated by an LLM and may be buggy, adversarial (if the
model is prompt-injected or the weights are tampered with), or simply wrong. The
sandbox is the containment boundary that lets us *run* such code to test it
without granting it the host's authority.

### Assets to protect

- The **host** running the Meta-Programmer (filesystem, processes, kernel).
- The **other Engram services** and the NATS bus (so generated code cannot
  forge messages — e.g. a `decision.*` or `policy.*` publish; see
  [ADR 0001](adr/0001-nats-authz.md)).
- **Secrets** reachable from the host (env vars, credentials, Docker socket).
- **Network-reachable systems** (no exfiltration, no SSRF, no lateral movement).

### In-scope threats

| # | Threat | Mitigation |
|---|---|---|
| T1 | Generated code opens network connections (exfiltration / C2 / SSRF) | `--network none` (`network_disabled=True`) — no interfaces but loopback. |
| T2 | Code writes to or tampers with the host/image filesystem | `--read-only` root FS; only `/tmp` is a size-capped tmpfs. Code is mounted **read-only** at `/sandbox`. |
| T3 | Privilege escalation (setuid, capability abuse) | `--cap-drop ALL` + `--security-opt no-new-privileges`; runs as non-root UID 10001. |
| T4 | Resource exhaustion / fork bombs / OOM the host | `--pids-limit`, `--memory`, `--cpus`, plus a wall-clock `--timeout` enforced by the manager (container killed on timeout). |
| T5 | Persistence across runs | `--rm` (auto-remove); no named volumes; tmpfs is discarded. |
| T6 | Access to the Docker socket / container breakout via daemon API | The socket is **never** mounted into the sandbox. Only the Meta-Programmer holds it. |
| T7 | Reaching other services / the bus | No network namespace connectivity (T1) means the sandbox cannot reach NATS at all. |

## 2. Hardening contract (image ↔ runtime)

The image is minimal; the **runtime flags** carry the isolation. Both must agree
or the boundary is weaker than documented. The flags below are applied in
`SandboxManager.run_tests()` and re-asserted by `sandbox/smoke/run_smoke.sh`:

| Control | Value | Source |
|---|---|---|
| Network | disabled | `network_disabled=True` / `--network none` |
| Root filesystem | read-only | `read_only=True` / `--read-only` |
| Writable path | `/tmp` tmpfs, 50 MB | `tmpfs={"/tmp": "size=50M"}` |
| Capabilities | all dropped | `cap_drop=["ALL"]` |
| Privilege escalation | blocked | `security_opt=["no-new-privileges"]` |
| User | non-root, UID 10001 (`sandbox`) | `sandbox/Dockerfile` |
| Code mount | read-only at `/sandbox` | `mode: "ro"` |
| Memory | 512 MB (`SANDBOX_MEMORY_LIMIT`) | `mem_limit` |
| CPU | ~50% of one core (`SANDBOX_CPU_QUOTA`) | `cpu_quota` |
| PIDs | 100 (`SANDBOX_MAX_PIDS`) | `pids_limit` |
| Wall clock | 30 s (`SANDBOX_TIMEOUT_SECONDS`) | `asyncio.wait_for` → `container.kill()` |
| Lifetime | auto-removed | `remove=True` / `--rm` |

## 3. Residual risk / out of scope

- **Kernel / container-runtime 0-days.** We rely on the Linux kernel and the
  container runtime for namespace/cgroup isolation. A kernel breakout defeats
  the sandbox; mitigated operationally by patching and, in higher-assurance
  deployments, by a VM-isolated runtime (gVisor / Kata) — not configured here.
- **CPU side-channels / timing.** Not addressed; the sandbox has no secrets and
  no network, limiting their value.
- **Supply chain of the base image and `pip` deps.** `python:3.12-slim` + a
  pinned `pytest`/`numpy` set; bumps go through `constraints.txt`.
- **Fail-closed deploy** (refuse to run if the image is missing or the runtime
  cannot apply a control) is tracked separately as **E1.3.2** and is *not* part
  of this issue. Today `SandboxManager` errors out if the image is absent, but
  the broader fail-closed guarantees land in E1.3.2.

## 4. Verification

`sandbox/smoke/run_smoke.sh` (run in CI, see `.github/workflows/test.yml`):

1. Builds the image via `docker compose --profile build-only build sandbox-base`.
2. Runs a **passing** generated test set inside the hardened container → expects
   exit 0.
3. Runs a **failing** generated test set → expects non-zero exit.
4. Asserts the network is unreachable (T1) and the root FS is read-only (T2)
   from inside the sandbox.
